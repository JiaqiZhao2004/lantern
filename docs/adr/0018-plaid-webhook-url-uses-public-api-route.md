# Plaid Webhook URL Uses Public API Route

Plaid webhook registration should use the app website route, `https://[lantern-public|lantern].royzhao.dev/api/v1/plaid/webhooks`, rather than the protected backend api hostname, `https://[lantern-public-api|lantern-api].royzhao.dev/api/v1/plaid/webhooks`. In production, Plaid calls CloudFront on the same public origin as the app, CloudFront forwards `/api/*` to the backend origin with its Cloudflare Access service-token headers, and the backend webhook route still authenticates the event with Plaid's webhook signature. This preserves the "protected backend origin" decision while allowing third-party server-to-server callbacks that cannot send Cloudflare Access credentials.

Each environment has exactly one configured Plaid webhook URL, provided by `PLAID_WEBHOOK_URL`, and every active InstitutionConnection's backing Plaid Item should match it. The URL is not stored on `institution_connections`: desired state lives in environment configuration, while actual state lives in Plaid. Deploy runs an explicit check-then-repair reconciliation after migrations and before runtime rollout, using Plaid's Item APIs to update missing or stale Item webhook URLs. We do this as a deploy gate rather than in FastAPI startup so process restarts stay free of hidden vendor mutations.

For local development, `PLAID_WEBHOOK_URL` should point at an HTTPS tunnel to the local backend, for example `https://<ngrok-domain>/api/v1/plaid/webhooks` with `ngrok http 8000`. The value must be present before creating a Plaid Link token, because current code registers the webhook URL through `link_token_create`; existing Plaid Items linked before the value was set are repaired by the same reconciliation command.
